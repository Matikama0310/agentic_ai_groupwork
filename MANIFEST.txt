WHAT THIS SYSTEM DOES
================================================================================

Insurance submissions flow through 5 phases:

  1. INGESTION     → Parse email + attachments
  2. EXTRACTION    → Extract data via OCR + LLM schema mapping
  3. ENRICHMENT    → Fetch external data in PARALLEL
                       • Internal claims history
                       • External credit/property risk (D&B, HazardHub)
                       • Web research (applicant website, reviews)
  4. ANALYSIS      → Validate against hard rules + assess risk + calculate premium
  5. OUTPUT        → Generate appropriate output based on decision

Possible Outcomes:
  ✅ QUOTED        → Generate quote email + PDF
  ❌ DECLINED      → Generate decline letter
  ⚠️  MISSING_INFO → Request missing documents
  🔄 MANUAL_REVIEW → Escalate for human review

ASSUMPTIONS & DECISIONS
================================================================================

ASSUMPTION-1: Mock External APIs (MVP)
  Current: All data sources return mock responses
  Rationale: Focus on orchestration logic, not data integration
  Migration: 2-hour swap to real APIs

ASSUMPTION-2: In-Memory State (MVP)
  Current: Dict-based state storage
  Rationale: No database setup required
  Migration: 1-hour swap to DynamoDB

ASSUMPTION-3: Sequential Enrichment
  Current: 3 data retrievers run sequentially
  Rationale: Lambda is single-threaded by design
  Migration: Use asyncio.gather() in production

ASSUMPTION-4: No LanGraph StateGraph (Yet)
  Current: Plain Python class with if/else control flow
  Rationale: Same functionality, simpler to debug
  Migration: Easy conversion to formal StateGraph

ASSUMPTION-5: No Real PDF/Email (MVP)
  Current: Placeholder URLs + draft emails
  Rationale: Focus on core orchestration
  Migration: 2-3 hours for ReportLab + SES
